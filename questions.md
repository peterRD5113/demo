# 業務邏輯疑問記錄

## 1. 風險規則匹配邏輯
- **問題**：當一個條款同時命中多個風險規則時（如"付款條件"和"違約責任"都涉及金額），如何處理？
- **我的理解**：應該顯示所有命中的規則，但按風險等級排序，最高風險的顏色優先顯示。
- **解決方式**：
  - 實現多標籤系統，一個條款可以有多個風險標註
  - 風險等級優先級：高風險（紅色）> 中風險（橙色）> 低風險（黃色）
  - UI 顯示時，條款背景色使用最高風險等級的顏色
  - 側邊欄詳情面板列出所有命中的規則及其建議

## 2. 版本管理的粒度
- **問題**：版本是按"整個文檔"保存，還是可以按"單個條款"保存？
- **我的理解**：應該按整個文檔保存版本，但變更摘要中列出具體修改的條款。
- **解決方式**：
  - 版本管理以整個文檔為單位
  - 每次保存版本時記錄：
    - 版本號（自動遞增）
    - 保存時間
    - 審閱人
    - 變更摘要（用戶輸入）
    - 變更條款列表（自動對比生成）
  - 回滾功能：直接恢復到選定的歷史版本（整個文檔）
  - 版本對比功能：顯示兩個版本之間的差異（高亮顯示變更的條款）

## 3. 多人協作場景
- **問題**：Prompt 提到"@同事待確認清單"，但又說"本地賬號登錄"，是否需要網絡同步？
- **我的理解**：@同事功能僅作為本地標記，不涉及實時通知，可能通過導出報告發送給同事。
- **解決方式**：
  - @同事功能實現為本地標記系統
  - 用戶可以在條款上添加 @標記，指定待確認人（從本地用戶列表選擇）
  - 待確認清單作為獨立視圖展示
  - 導出報告時包含所有 @標記和待確認事項
  - 不實現實時通知或網絡同步功能
  - 未來可擴展：通過導出的報告文件在團隊間傳遞

## 4. 敏感信息脫敏範圍
- **問題**：哪些信息算"敏感信息"？金額、人名、公司名、地址？
- **我的理解**：應該提供可配置的脫敏規則（如身份證號、銀行賬號、金額），用戶可自定義。
- **解決方式**：
  - 內置預設脫敏規則：
    - 身份證號：顯示前 3 位和後 4 位，中間用 * 代替
    - 銀行賬號：顯示後 4 位，其餘用 * 代替
    - 手機號：顯示前 3 位和後 4 位，中間用 * 代替
    - 金額：顯示為 "***元" 或保留量級（如 "10萬+" ）
    - 人名：顯示姓氏，名字用 * 代替
    - 公司名：顯示簡稱或用 "某公司" 代替
  - 用戶可自定義脫敏規則（正則表達式 + 替換模式）
  - 脫敏採用可逆映射：
    - 原始數據加密存儲在本地數據庫
    - 脫敏後的文檔顯示替換後的內容
    - 用戶可選擇"顯示原文"或"脫敏預覽"模式
  - 導出時可選擇是否包含敏感信息

## 5. 風險等級判定標準
- **問題**：風險等級（高/中/低）如何判定？是規則預設還是用戶評估？
- **我的理解**：規則庫預設基礎風險等級，用戶可在審閱時調整。
- **解決方式**：
  - 規則庫中每條規則預設風險等級：
    - 高風險：涉及重大財務損失、法律責任、核心權益的條款
      - 例：無上限違約金、自動續約、排他性條款、管轄地不利
    - 中風險：可能造成不便或額外成本的條款
      - 例：付款條件不明確、保密期限過長、通知期限過短
    - 低風險：需要注意但影響較小的條款
      - 例：格式不規範、用詞不嚴謹、參考標準過時
  - 用戶可在審閱時手動調整風險等級：
    - 右鍵菜單或側邊欄提供"調整風險等級"選項
    - 調整後的等級僅對當前文檔生效
    - 可選擇"應用到規則庫"，更新全局規則
  - 風險等級影響：
    - 顏色標註（紅/橙/黃）
    - 審閱報告中的排序
    - 待確認清單的優先級

## 6. 文檔結構識別邏輯
- **問題**：如何自動生成條款編號？不同合同的結構可能差異很大（有的用數字編號，有的用中文序號，有的用章節標題）。
- **我的理解**：應該提供多種編號模式，並支持手動調整。
- **解決方式**：
  - 自動識別文檔結構：
    - 檢測現有編號模式（1.1.1、一、（一）、第一條等）
    - 識別標題層級（字體大小、加粗、縮進）
    - 識別段落結構（空行、縮進）
  - 提供多種編號模式：
    - 數字模式：1、1.1、1.1.1
    - 中文模式：一、（一）、1
    - 法律模式：第一條、第一款、第一項
    - 自定義模式：用戶可定義編號規則
  - 支持手動調整：
    - 用戶可拖拽調整條款順序
    - 可合併或拆分條款
    - 可手動修改編號
  - 自動生成目錄：
    - 基於條款編號和標題生成
    - 支持多級目錄（最多 3 級）
    - 點擊目錄項可快速跳轉

## 7. 常用條款模板管理
- **問題**：常用條款模板如何組織？是否需要分類？如何快速插入？
- **我的理解**：應該提供分類管理，並支持快速搜索和插入。
- **解決方式**：
  - 模板分類：
    - 內置分類：付款條款、違約責任、保密條款、爭議解決、終止條款等
    - 用戶可自定義分類
  - 模板管理：
    - 每個模板包含：標題、內容、適用場景、風險提示
    - 支持模板的增刪改查
    - 支持從現有條款創建模板
  - 快速插入：
    - 側邊欄提供模板庫面板
    - 支持關鍵詞搜索
    - 拖拽或點擊插入到指定位置
    - 插入後可直接編輯
  - 模板變量：
    - 支持變量佔位符（如 {{甲方名稱}}、{{金額}}）
    - 插入時彈出表單填寫變量
    - 自動替換佔位符

## 8. 性能優化策略
- **問題**：如何確保 10MB 文檔在 8 秒內打開？
- **我的理解**：需要採用分段加載、索引優化、緩存機制。
- **解決方式**：
  - 文檔解析優化：
    - 使用 Worker 線程進行文檔解析，避免阻塞 UI
    - 分段解析：先解析前 100 個條款，其餘後台加載
    - 建立條款索引：快速定位和搜索
  - 渲染優化：
    - 虛擬滾動：只渲染可見區域的條款
    - 延遲加載：圖片、表格等複雜元素延遲渲染
    - 增量渲染：分批渲染條款，避免一次性渲染卡頓
  - 緩存機制：
    - 解析結果緩存到本地數據庫
    - 文檔指紋（MD5）檢測變更
    - 未變更的文檔直接從緩存加載
  - 數據庫優化：
    - 為常用查詢字段建立索引
    - 使用事務批量操作
    - 定期清理過期緩存

## 9. 導出格式保真度
- **問題**：導出的 docx 和 PDF 如何保證與原文檔格式一致？
- **我的理解**：應該保留原文檔的格式信息，並在導出時還原。
- **解決方式**：
  - 格式信息保留：
    - 解析時保存字體、字號、顏色、對齊方式等格式信息
    - 保存段落樣式、列表樣式、表格樣式
    - 保存頁眉頁腳、頁碼、水印等元素
  - 導出 docx：
    - 使用 docx 庫重建文檔結構
    - 還原所有格式信息
    - 添加修訂痕跡（Track Changes）
    - 添加批註（Comments）
  - 導出 PDF：
    - 使用 pdf-lib 生成 PDF
    - 保留原文檔排版
    - 添加批註標記（Annotations）
    - 支持高亮、下劃線、刪除線等標記
  - 審閱報告：
    - 生成獨立的 PDF 報告
    - 包含風險摘要、條款清單、建議修改
    - 支持自定義報告模板

## 10. 登錄失敗鎖定機制
- **問題**：連續登錄失敗 5 次鎖定 10 分鐘，如何實現？是否需要考慮暴力破解？
- **我的理解**：需要記錄失敗次數和鎖定時間，並提供解鎖機制。
- **解決方式**：
  - 失敗計數：
    - 記錄每個賬號的登錄失敗次數
    - 記錄失敗時間戳
    - 成功登錄後重置計數
  - 鎖定機制：
    - 連續失敗 5 次後鎖定賬號
    - 記錄鎖定開始時間
    - 10 分鐘後自動解鎖
    - 鎖定期間顯示剩餘時間
  - 安全增強：
    - 密碼使用 bcrypt 加密存儲
    - 登錄日誌記錄（時間、IP、結果）
    - 支持管理員手動解鎖
    - 可配置失敗次數和鎖定時長
  - 防暴力破解：
    - 登錄請求添加延遲（每次失敗後延遲遞增）
    - 可選：添加圖形驗證碼（失敗 3 次後）
    - 可選：支持雙因素認證（未來擴展）
