# 阶段2实现计划 - 详细分析

## 🎯 目标
在 DocumentReviewPage 中实现对照修订功能，让用户可以对比当前编辑和最新保存版本

## 🤔 深度思考

### 1. 核心需求分析

#### 用户场景
```
用户修改了一些条款
    ↓
想要查看与上次保存版本的差异
    ↓
点击"对照修订"按钮
    ↓
页面切换到双栏视图
    ↓
左侧：最新保存版本（只读）
右侧：当前编辑内容（可编辑）
    ↓
可以同步滚动查看
    ↓
可以继续编辑右侧内容
    ↓
点击"保存版本"创建新快照
```

#### 关键问题
1. **如何获取最新版本？**
   - 使用 `window.electronAPI.version.getLatest(documentId, userId)`
   - 返回最新版本的元数据

2. **如何获取版本的条款？**
   - 使用 `window.electronAPI.version.getClauses(versionId, userId)`
   - 返回该版本的所有条款快照

3. **如何切换视图？**
   - 使用状态 `isComparisonMode` 控制
   - true: 显示双栏
   - false: 显示单栏（原有布局）

4. **如何同步滚动？**
   - 使用 useRef 获取两个滚动容器
   - 监听 onScroll 事件
   - 使用防抖避免循环触发

5. **目录如何处理？**
   - 添加"收起目录"按钮
   - 使用状态 `sidebarCollapsed` 控制显示/隐藏
   - 对照模式下可能需要更多空间

### 3. 实现难点分析

#### 难点1：条款数据结构差异
**问题**：
- 当前条款：从 `clauses` 状态获取，包含 level 字段
- 版本条款：从 API 获取，可能结构不同

**解决方案**：
- 查看 API 返回的数据结构
- 确保版本条款也有 level 字段
- 统一数据格式

#### 难点2：滚动同步
**问题**：
- 两个容器互相监听会导致循环触发
- 需要防抖机制

**解决方案**：
- 使用 `isSyncing` 标志位
- 滚动时设置标志，防止另一侧触发
- 使用 requestAnimationFrame 优化性能

#### 难点3：条款渲染逻辑复杂
**问题**：
- 现有代码中条款渲染有多种情况：
  - level === -1: 文档标题
  - level === 0: 章节
  - level > 0: 条款
- 需要在双栏模式中复制这些逻辑

**解决方案**：
- 提取条款渲染为独立函数
- 或者在双栏模式中复制渲染逻辑
- 确保左右两侧渲染一致

#### 难点4：编辑功能保持
**问题**：
- 右侧需要保持可编辑
- 左侧需要禁用编辑
- 批注面板需要正常工作

**解决方案**：
- ClauseItem 组件已支持 editable 属性
- 左侧：editable={false}
- 右侧：editable={true}
- 批注面板保持原有逻辑

### 7. 关键决策

#### 决策1：是否提取条款渲染为独立组件？
**选项A**：提取为 ClauseRenderer 组件
- 优点：代码复用，逻辑清晰
- 缺点：需要额外创建文件，增加复杂度

**选项B**：在双栏模式中复制渲染逻辑
- 优点：简单直接，不需要新文件
- 缺点：代码重复

**决定**：选择 B，因为：
- 渲染逻辑不复杂
- 避免过度工程化
- 保持文件结构简单

#### 决策2：目录在对照模式下如何处理？
**选项A**：对照模式下自动隐藏目录
- 优点：更多空间给双栏
- 缺点：用户可能需要目录导航

**选项B**：保持目录可见，让用户选择
- 优点：灵活性高
- 缺点：可能空间不够

**决定**：选择 B，因为：
- 用户可以自己决定
- 添加"收起目录"按钮
- 更符合用户习惯

#### 决策3：是否需要差异高亮？
**当前阶段**：不实现
**原因**：
- 增加复杂度
- 需要文本对比算法
- 可以作为后续优化

**未来优化**：
- 使用 diff 算法对比内容
- 高亮显示修改部分
- 不同颜色表示增删改
