# 律影合同審閱系統 - 項目總結

## 📋 項目分析總結

我已經完成了對「律影合同審閱與條款風險標註桌面工作台」的深度需求分析和規劃。以下是關鍵要點：

## 1️⃣ 項目類型判斷

根據 `.cursor/rules/05-project-type-classification.mdc` 規範：

- **項目類型**：桌面端應用（Desktop）
- **容器化要求**：❌ 不需要 Docker
- **啟動方式**：雙擊可執行文件
- **技術選型**：TypeScript + Tauri（優先）或 Electron + React + Ant Design

## 2️⃣ 核心功能拆解

### 文檔管理
- 支持 docx/pdf/txt 三種格式導入
- 自動生成目錄與條款編號（多級嵌套）
- 提供「逐條審閱」和「對照修訂」兩種視圖
- 虛擬滾動優化大文檔性能

### 風險識別引擎
- 可配置的風險規則庫（付款條件、違約責任、管轄地、保密期限、自動續約）
- 關鍵詞 + 正則表達式 + 置信度評分
- 顏色標註 + 建議措辭
- 支持用戶自定義規則

### 協作功能
- 批註系統（評論、建議、問題）
- @同事待確認清單
- 常用條款模板庫（支持變量替換）

### 版本管理
- 自動/手動版本保存
- 完整快照 + 增量 diff
- 版本對比與回滾
- 版本樹結構（支持分支）

### 導出功能
- 帶批註的 PDF
- 修訂痕跡的 docx（Track Changes）
- 審閱摘要報告（風險統計、待確認問題、建議修改清單）

### 安全機制
- 本地賬號登錄（bcrypt 加密）
- 連續失敗 5 次鎖定 10 分鐘
- 項目訪問密碼
- 敏感信息脫敏（身份證、手機號、銀行賬號）
- 一鍵清理緩存

### 性能要求
- 首次打開 5 秒內可用
- 10MB 文檔打開不超過 8 秒
- 離線可用

## 3️⃣ 隱含約束識別

根據 `.cursor/rules/00-core-standards.mdc` 的「需求理解深度」要求，我識別了以下隱含約束：

### 數據一致性
- 版本回滾後不刪除後續版本（創建新分支）
- 文檔解析失敗時不保存不完整數據
- 風險掃描結果與文檔內容保持同步

### 並發控制
- 文件級鎖定（.lock 文件）防止多人同時編輯
- 版本號機制檢測衝突

### 用戶體驗
- 大文檔使用 Web Worker 解析（不阻塞 UI）
- 虛擬滾動僅渲染可見區域
- SplashScreen 提升感知速度

### 安全邊界
- 密碼不可找回（僅支持重置）
- 日誌不記錄敏感信息（token、密碼）
- 導出文件可選水印

### 數據隔離
- 用戶只能訪問自己創建的項目
- 項目密碼獨立加密存儲
- 操作審計可追溯

## 5️⃣ 技術架構亮點

### 分層架構
```
前端層（React + TypeScript）
    ↓
業務邏輯層（Services）
    ↓
數據層（SQLite + FileSystem）
```

### 核心技術選型
- **桌面框架**：Tauri（體積小、性能好）
- **UI 框架**：Ant Design（中文友好）
- **文檔解析**：mammoth.js（docx）+ pdf-parse（pdf）
- **文檔生成**：docx（修訂痕跡）+ pdf-lib（批註）
- **數據庫**：better-sqlite3（同步 API、高性能）
- **虛擬滾動**：react-window（大文檔優化）

### 性能優化
- Web Worker 文檔解析
- 虛擬滾動（僅渲染可見區域）
- 條款索引緩存
- 延遲加載非核心模塊
- SplashScreen 提升感知速度
